## 1. 일자별 작업 로그 (백업 체계) ★★★★★

> 핵심 요구: **작업 오류로 되돌릴 때 "○월 ○일 상태로 복구"가 가능한 수준**으로 매 작업을 기록.

### 1.1 작성 규칙
- 위치: `docs/작업로그/YYYY-MM-DD_작업로그.md`
- **하루에 한 파일**, 그날 한 모든 작업을 누적 기재 (덮어쓰지 말고 이어쓰기)
- 작업이 끝날 때마다 즉시 기록 (몰아서 X)
- 이전 로그는 **절대 삭제·수정 금지** (백업 원본이므로)

### 1.2 로그에 반드시 들어갈 항목

```markdown
# 2026-MM-DD 작업로그

## [작업 N] 작업 제목 (시각)
- **무엇을**: 변경한 기능/파일
- **왜**: 이 작업을 한 이유 (사용자 요청 추적)
- **어떻게**: 구체적 변경 내용 (파일명·함수·핵심 로직)
- **변경 파일 목록**: path 나열
- **이전 상태 → 이후 상태**: 되돌릴 수 있게 before/after 명시
- **커밋 해시**: (git 사용 시) 복구 기준점
- **검증**: 어떤 시나리오로 동작 확인했나
- **남은 것 / 다음 작업**: 이어서 할 일
```

### 1.3 복구 시나리오 대응
- 사용자가 "○월 ○일로 되돌리자"라고 하면:
  1. 해당 날짜 로그의 **커밋 해시**로 git 복원
  2. 그 이후 로그들을 역순으로 확인해 무엇이 사라지는지 사용자에게 보고
  3. 복원 전 현재 상태를 새 로그에 백업 기록
- git 커밋 해시를 매 작업 로그에 남기는 이유가 이것. **해시 = 복구 좌표**.

## 2. 오류 반복 방지 기준 (누적 관리) ★★★★★

> 핵심 요구: 수정 작업 중 같은 오류가 반복되지 않도록 기준을 잡고, 작업 시 무조건 먼저 참조.

### 2.1 두 개의 기준 문서

| 파일 | 역할 | 갱신 시점 |
|------|------|-----------|
| `docs/기준/00_금지사항_체크리스트.md` | 하지 말아야 할 것 (실수 → 규칙화) | 오류 발생 때마다 |
| `docs/기준/01_결정사항_누적.md` | 이미 확정된 설계 결정 | 결정 내릴 때마다 |

### 2.2 오류 → 규칙 전환 절차 (재발 방지의 핵심)
같은 실수를 두 번 하지 않기 위해, 오류가 나면 **반드시 규칙으로 박제**한다:

```
오류 발생
  → 원인 파악
  → 00_금지사항_체크리스트.md 에 한 줄 추가
     "[YYYY-MM-DD] (상황) 시 (행동) 금지 — 이유: ..."
  → 다음 작업부터 이 체크리스트를 먼저 보고 위배 여부 확인
```

> 이게 "기준을 잡아 오류 반복을 막는" 메커니즘. 사람 기억에 의존하지 않고 문서가 기억한다.

### 2.3 결정사항 누적의 이유
계획서 v1.0~v2.5에서 보듯, 결정은 계속 바뀌고 쌓인다. "이미 이렇게 하기로 했는데 왜 또?"를 막으려면 결정을 한곳에 누적:

```markdown
[YYYY-MM-DD] 파일 저장은 구글 드라이브 백엔드, 외부 공유 금지
[YYYY-MM-DD] 습관 = 반복 켜진 할 일로 통합 (별도 테이블 X)
[YYYY-MM-DD] 자동 재생성은 완료 트리거 + 생성일/안내일 분리(리드타임)
...
```

## 3. 큰 작업은 Sprint Plan 먼저

**1일 이상 걸리는 작업**(새 기능, 큰 리팩토링)은 코드 작성 전 plan부터:

1. `docs/sprints/_template.md` 복사 → `docs/sprints/NN-기능명.md`
2. plan 작성: 목표 / 완료기준 / 영향 파일 / 단계별 PR / 리스크
3. **사용자에게 보여주고 승인 받기** ← 승인 전 구현 금지
4. 승인 후 구현 시작
5. PR/작업 단위로 머지 + plan 체크박스 갱신 + **작업로그 기록**
6. Sprint 종료 시 계획서(v2.5) 해당 부분과 동기화

## 4. 작은 작업은 Todo

1시간 이하 작업·버그 수정·UI 미세 조정은 plan 문서 없이 진행 OK. 단 **작업로그 기록은 생략 불가**.

---

## 5. 코드 변경 원칙 (Karpathy 4-rule)

1. **확신 없으면 묻기** — 가정을 조용히 깔지 말 것. 해석이 여러 개면 골라달라고 할 것.
2. **요청한 것만 변경** — 인접 코드 "개선" 금지, 스타일 통일 금지, 조용한 리팩터 금지. 모든 변경 라인은 사용자 요청에 trace 가능해야 함.
3. **테스트 가능한 목표** — "동작하게"가 아니라 "이 시나리오 통과"로 정의. 다단계 작업은 step→verify로.
4. **작은 단위로** — 한 번에 하나씩, 검증하고 다음으로.

## 6-B. 깃 저장소 운영

- **저장소**: https://github.com/ssm741031/crm-20260530.git
- **메인 브랜치**: `main` — 항상 동작하는 상태 유지
- **작업 흐름**:
  1. 기능 작업은 브랜치 분리 권장 (`feat/기능명`)
  2. 작업 단위로 커밋 (6장 메시지 규칙)
  3. **커밋 직후 해시를 작업로그에 기록** (1.2) — 복구 좌표
  4. 검증 후 `main` 병합
- **복구와의 연결**: "○월 ○일로 되돌리자" → 해당 날짜 작업로그의 커밋 해시로 `git checkout`/`git revert`. 그래서 매 커밋 해시 기록이 필수.
- **금지** (Stop the line, 7장과 연동):
  - `git push --force` to `main` 금지
  - 커밋에 비밀번호·시크릿·고객 개인정보 포함 금지 (`.gitignore`로 `.env`·키파일 제외)
  - 첨부 파일 실물(청약서·증권 등)을 저장소에 커밋 금지 — 파일은 구글 드라이브 백엔드(계획서 2.1-d)
- **초기 세팅 시**: `.gitignore`에 `.env`, 자격증명, 업로드 파일 경로 포함 확인

## 7. Stop the line — 즉시 멈추고 사용자에게 묻기
