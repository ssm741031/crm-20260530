# Sprint 01: 프로젝트 기초 + 인증·권한

> 작성일: 2026-05-30 · 상태: 승인대기
> 관련 계획서 섹션: 사내CRM_개발계획서_v2.5.md §2.1(직원), §8(보안)
> 저장소: https://github.com/ssm741031/crm-20260530.git

## 목표
프로젝트 뼈대를 세우고, 20인이 **로그인해서 권한(대표/팀장/팀원)에 맞게 접근**하는 최소 기반을 만든다. 이후 모든 기능이 이 위에 얹힌다.

## 왜 이것부터 (추천 근거)
- 인증·권한은 **모든 화면의 전제**다. 고객 데이터를 다루므로 "누가 로그인했고 무엇을 볼 수 있나"가 없으면 어떤 기능도 안전하게 못 만든다.
- 계획서 8장(보안)의 권한 통제·서버 보안이 여기서 시작된다.

## 완료 기준 (테스트 가능하게)
- [ ] 저장소 클론 → 로컬에서 프론트·백엔드·DB가 실행된다
- [ ] 직원이 ID/비밀번호로 **로그인/로그아웃**할 수 있다
- [ ] 비밀번호는 **해시로 저장**된다 (평문 DB에 없음 — 확인)
- [ ] 로그인하면 권한(대표/팀장/팀원)이 식별된다
- [ ] 권한별로 접근 범위가 다르다: 대표=전체 / 팀장=팀 / 팀원=본인 (빈 화면으로라도 분기 확인)
- [ ] 서버(API)에서도 권한을 **재검증**한다 (프론트만 숨기는 게 아님)
- [ ] HTTPS로 접속된다 (고보험 서버)

## 영향 받는 파일 (예상)
- `backend/` — FastAPI 앱 초기화, User 모델, 인증 라우터, 권한 미들웨어
- `frontend/` — React 앱 초기화, 로그인 화면, 권한 컨텍스트, 라우팅
- `backend/db/` — PostgreSQL User 테이블 (이름·로그인ID·권한·비밀번호해시)
- `.gitignore` — `.env`·자격증명 제외
- `.env.example` — DB 접속·시크릿 키 양식 (실제 값은 커밋 금지)

## 단계별 작업 (step → verify)
1. [ ] 저장소 초기화 + `.gitignore` + 모노레포 구조(frontend/backend) → 검증: 빈 앱 실행
2. [ ] PostgreSQL 연결 + User 모델 + 마이그레이션 → 검증: 테이블 생성 확인
3. [ ] 회원 1명 시드(대표 계정) — **비밀번호는 사용자가 직접 입력해 생성** → 검증: 해시 저장 확인
4. [ ] 로그인 API (해시 검증, 세션/JWT 발급) → 검증: 올바른/틀린 비번 분기
5. [ ] 프론트 로그인 화면 + 로그인 상태 유지 → 검증: 새로고침 후 유지
6. [ ] 권한 미들웨어 (API에서 역할 검증) → 검증: 팀원이 전체 데이터 요청 시 차단
7. [ ] 고보험 서버 배포 + HTTPS 확인 → 검증: https 접속

## 리스크 / 주의 (금지사항 체크리스트 연동)
- **비밀번호 평문 금지** — 해시 필수 (금지사항)
- **계정 생성은 사용자가** — Claude가 임의 계정 생성·비밀번호 설정 금지
- **고보험 서버 선점검** — HTTPS 여부, 자원 여유, 기존 서비스 포트·DB 충돌 (선결 과제)
- `.env`·시크릿 커밋 금지 — `.gitignore` 먼저
- 권한은 프론트뿐 아니라 **서버에서 재검증** (계획서 8.2)

## 자기비판 (이 계획의 약점)
- 기존 고보험 서버 상태를 모르는 채로 7단계를 잡았다. **서버 점검 결과에 따라 배포 단계가 크게 바뀔 수 있다.** 점검 먼저 하고 7단계를 구체화하는 게 안전.
- 인증 방식(세션 vs JWT)을 아직 안 정했다. 20인 내부용이면 세션이 단순하나, 모바일 PWA 고려 시 JWT가 유리. 구현 전 택1 필요.

---
**선결 과제 (구현 전 확인)**:
- ☐ 고보험 서버 HTTPS 적용 여부
- ☐ 고보험 서버 여유 자원 (CPU/메모리/디스크)
- ☐ 기존 고보험 서비스와 포트·DB 충돌 여부
- ☐ 인증 방식 결정 (세션 / JWT)
