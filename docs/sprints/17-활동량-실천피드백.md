# Sprint 17: 활동량 · 실천 피드백 화면

> 작성일: 2026-06-01 · 상태: 진행중 (2026-06-01 승인됨)
> 관련 계획서: §6(실천 피드백·활동량 누적), §6.1(일일 처리현황), §6.2(누적 활동량)
> 담당: 화면·구동로직 = 작업자 / 서버 = 이팀장

## 목표
지금까지 쌓인 데이터(할 일 완료 여부·날짜, 파이프라인 단계)로 **"계획 대비 실천"을 한눈에** 보는 활동량 화면을 만든다.

## 범위 (이번 스프린트)
포함 (지금 데이터로 가능):
- **오늘 처리 현황**: 오늘 마감 할 일 중 완료 비율 (계획서 §6.1)
- **기간 달성률**: 최근 7일/30일 완료 건수·달성률
- **카테고리별 완료 분포**: 어느 활동에 실천이 몰리는가
- **습관 streak 요약**: 반복 항목 연속 달성 (이미 있는 streak 활용)
- 권한 범위 내 데이터만 (currentUser 기준, 대표=전체)

제외 (보류/후속):
- **단계별 병목 분석**(§3.3): SLA 기준시간과 엮임 → 보류 항목과 함께 후순위
- 팀원별 활동량 비교: 팀 모델 부재(권한 1차 한계) → 후속
- 일자별 완료 이력 정밀 집계: 서버(완료 시각 누적 이력) 필요

## 왜 지금
- 영업 운영의 "자기 점검 루프"(계획 → 실천 → 피드백) 핵심.
- 이미 있는 데이터로 서버 없이 구현 가능 (목 우선 전략에 부합).

## 리스크 / 주의 (금지사항 연동)
- **데이터 착시 경고**: 목 데이터라 통계가 적음 — "목 데이터 기준" 안내. 반복 항목 자동전개는 서버 후라 streak/달성률이 실제와 다를 수 있음 명시
- 권한: 통계도 권한 우회 통로가 되면 안 됨(§8) — canViewTask로 먼저 필터 후 집계
- 0건/빈 상태 깨짐 방지 (나눗셈 0 division 가드)
- 단계 병목·팀 비교는 이번에 넣지 않음(가짜 수치 금지)
- OneDrive Write 후 즉시 검증 / HMR import 먼저

## 자기비판 (이 계획의 약점)
- 목 데이터가 적어 통계가 빈약하게 보일 수 있음 — 구조·계산 검증이 목적, 실제 인사이트는 운영 데이터 쌓인 뒤.
- "계획 대비"의 '계획'을 오늘 마감 할 일로 정의 — 더 정교한 계획(일일 목표 설정)은 후속.
- 완료 시각 이력이 없어 "어제 몇 건 완료"는 doneAt 기반 근사. 서버 완료이력 붙으면 정확해짐.
