> 작성일: 2026-05-31 · 상태: 완료 (2026-05-31 구현·검증 완료, 커밋 c9c6b3a)
> 관련 계획서: §2.4-b(안내 기록 Notice Log), §2.4-c(수신동의 Consent), §8.5(마스킹)
> 담당: 화면·구동로직 = 작업자 / 서버 = 이팀장

## 목표
두 가지 수동 기록 UI를 만든다. Phase 3 자동발송의 기반 데이터:
1. **수신동의** — 고객별 문자/카톡 수신동의(동의·거부·미확인) + 동의일을 **편집**
2. **안내 기록** — 청약 단계에서 "문자/카톡/전화로 안내함"을 체크·기록 (수단·일시·메모)

## 왜 지금 / 왜 같이
- 둘 다 Phase 1 "사람이 보내고 기록만" → Phase 3 자동발송 자격판단 근거 (계획서 §7 안내방식 전환).
- 수신동의 타입(`Consent`)은 이미 있으나 **입력 UI가 없음**(고객 화면 읽기전용).
- 안내 기록은 **타입조차 없음** — 신규. 파이프라인 단계에 붙는 구조라 PipelinePage와 연결.

## 범위
포함:
- 고객 수신동의 편집(고객 화면에서 인라인 또는 작은 폼)
- `NoticeLog` 타입 신규 + 단계별 안내 기록 추가/조회 UI(파이프라인 단계)
제외(후속):
- 자동 발송(Phase 3), 안내 템플릿 자동화, 동의 이력(변경 히스토리)

## 리스크 / 주의 (금지사항 연동)
- **개인정보**: 안내 메모에 고객 민감정보 들어갈 수 있음 — 로그 출력 금지, 화면 표시만. 연락처 노출 시 마스킹(`utils/mask.ts`)
- **수신동의는 법적 근거 데이터**(정보통신망법) — 함부로 기본값 "동의"로 만들지 말 것. 신규/미설정은 **"미확인"** 유지
- 안내 일시는 **시스템이 자동 기록**(수동 입력 금지 — 위변조 방지). 작성자도 자동(현재 mock currentUser)
- 목 상태라 새로고침 시 초기화(안내 유지)
- OneDrive Write 후 즉시 검증 / HMR import 먼저 (금지사항)

## 자기비판 (이 계획의 약점)
- 고객 "상세 화면"이 없어 수신동의 편집을 목록에 인라인으로 붙이면 좁을 수 있음 → 작은 모달/펼침으로 완화. 고객 상세 화면은 별도 후속.
- NoticeLog를 Stage에 넣을지 Pipeline 배열로 둘지 — 데이터 정합상 Pipeline.noticeLogs[] (stageNo 필드로 단계 구분)가 단순. 단계 삭제 개념 없으므로 충분.
- 동의 변경 이력(언제 누가 바꿨나)은 1차 미기록 — 감사 필요 시 후속.
