## 목표
고객별 청약을 **상품 5종 템플릿**으로 시작하고, **단계 완료 → 다음 단계 마감 자동 산출**로 진행한다.
마감 초과를 표시하고, **연장(사유·기간)**으로 마감을 재설정한다.

## 동작 원리 (계획서 §3.1)
- 단계 완료 시 다음 단계 마감 = 완료시점 + 기준시간(템플릿 일수). 마감 시계는 "현재 단계"에만.
- 자동차 갱신은 **만기일 역산**: 1단계 만기 D-60, 2단계 D-30, 3단계 고객요청일(수동), 4단계 앞단계+1.
- 만기일 = 절대 마감(하드 데드라인) → 강조 표시.

## 리스크 / 주의 (금지사항 연동)
- 실제 알림/푸시는 가짜로 만들지 않는다 — "지연" 상태·강조 표시만, 발송은 서버(§4)
- 만기일 = 하드 데드라인 → 넘기면 무보험 공백 위험을 강조 표기(계획서 §2.5)
- 연장은 "그냥 미루기" 아님 → 사유·기간 기록 필수(§3.2). 사유 없는 연장 금지
- 기준시간은 템플릿 초기값(코드 고정 아님 원칙) — 이번엔 적용만, 편집은 후속임을 명시
- 고객 개인정보는 화면 표기만(이름), 로깅·외부전송 금지(금지사항)
- PipelinePage CRUD가 타 페이지와 패턴 중복 → 복제 허용, 추후 훅 추출 검토(결정 후보)
- OneDrive Write 후 즉시 검증 / preview 상호작용 순차 (금지사항 2026-05-30)
- 요청 외 인접 리팩터 금지 (Karpathy 2)

## 자기비판 (이 계획의 약점)
- 단계 마감 산출을 "완료시점+일수"로 즉시 계산하는데, 실제 운영은 영업시간/주말 고려가 필요할 수 있음 → 이번엔 단순 일수 가산, 영업일 계산은 후속(서버 정책).
- 자동차갱신 3단계(고객요청일)는 수동 입력 — 입력 전엔 마감 미정(null) 처리, UI에 "요청일 입력 필요" 안내.
- SLA 편집·병목분석을 빼서 "운영 튜닝" 가치는 후속에 의존. 이번은 진행 파이프라인의 뼈대·연장까지가 목표.
- delays를 Pipeline에 배열로 두는 건 임시(서버는 별도 테이블 §2.4) — 모델 확정 시 정리(결정 보류 연동).
