## 자기비판 (이 계획의 약점)
- mock pw "1234" 통일은 진짜 보안 효과 0 — 의도된 dev 편의용. 실수로 운영에 남으면 문제
- localStorage 저장은 XSS 시 자격증명 노출 가능 — 진짜 인증은 httpOnly 쿠키 권장 (이팀장 결정)
- "잘못된 id/pw" 분기에서 timing attack 방지 없음 (mock이라 의미 없음)
- 다중 탭 로그아웃 동기화 (storage 이벤트) 1차 미구현 — 다른 탭이 로그아웃해도 본 탭은 살아있음. 후속
- 비밀번호 변경/찾기 등 부가 기능은 별도 sprint
- 백엔드 완성 시 인증 방식(세션 vs JWT, 쿠키 vs Authorization 헤더)에 따라 api/auth.ts 일부 조정 필요할 수 있음
