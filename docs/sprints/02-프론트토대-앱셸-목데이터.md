## 목표
서버(이팀장 Sprint 01)와 **독립적으로** 화면·로직을 개발할 수 있는 프론트 토대를 만든다.
React + Vite + TypeScript 앱 셸 + 라우팅 + 반응형 레이아웃 + **목 데이터 레이어**까지.
이후 모든 화면(카테고리·할 일·캘린더·고객·파이프라인)이 이 위에 얹힌다.
서버가 완성되면 목 레이어만 진짜 API로 교체한다.

## 왜 이것부터 (추천 근거)
- 서버가 아직 없으므로(이팀장 Sprint 01 승인대기), 목 데이터로 분리하지 않으면 화면 개발이 시작조차 못 한다.
- 앱 셸·라우팅·반응형·디자인 토큰은 **모든 화면의 공통 토대**라 한 번 잘 잡으면 재사용된다.
- 목 데이터 레이어를 "API 계약"과 같은 모양으로 만들면, 나중에 교체가 한 파일 수정으로 끝난다 → 이팀장과의 접점 최소화.

## 자기비판 (이 계획의 약점)
- 이팀장과 **API 계약을 아직 문서로 확정하지 않았다.** §2를 잠정 기준으로 가되, 인증 방식(세션/JWT)·파일 업로드 경유 방식이 정해지면 api 레이어를 한 번 조정해야 할 수 있다.
