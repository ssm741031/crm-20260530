## 목표
반복(매주/매월/매년) + 자동재생성이 켜진 할 일을 **완료**하면, 다음 주기 할 일이
**만기일 역산**으로 자동 생성된다. 생성일·안내일·만기일 3단 분리를 폼에서 설정·미리보기한다.

## 핵심 개념 (계획서 §5-B.3)
```
[투두 생성] ──리드타임──> [안내 목표] ──────> [만기일]
 만기-(안내+리드)일 전     만기-안내일 전        하드 데드라인(endDate)
```
- 안내 목표: 만기 N일 전 (30/60/90 + 직접 입력)
- 리드타임: 안내일보다 N일 먼저 생성 (1/3/7/14 + 직접 입력)
- 예) 자동차 갱신: 안내 60일 전 + 리드타임 7일 → 만기 **67일 전** 생성

## 범위 (사용자 결정 2026-05-30: "폼 입력 + 완료시 즉시 재생성")
포함:
- Task 타입 확장: `regenNoticeDaysBefore`, `regenLeadDays` (만기일 = 기존 endDate 사용)
- 폼: autoRegen 토글(**매주/매월/매년에서만** 활성), 안내목표·리드타임 입력, 3단 날짜 미리보기
- 완료 토글 시 자동재생성: 다음 주기 만기일(endDate +1주/월/년) 계산 → 다음 할 일 즉시 생성(목 목록에 등장), streak +1
- 날짜 역산 유틸(주/월/년 더하기, 안내일·생성일 계산)

**제외(다음/서버)**:
- **리드타임만큼 미리 생성하는 스케줄러** → 서버·Phase (이팀장). 이번엔 완료 시 즉시 생성으로 로직만 검증
- 빼앗긴 고객 처리 3종(§5-B.4) → 운영 중 결정(결정사항 누적). 자리표시 문구만
- 재생성 미완료 N회 중단 안전장치 → 서버·후순위 (이번엔 무한루프 없음: 완료해야만 1건 생성)
- 습관 뷰·캘린더 → 별도

## 자기비판 (이 계획의 약점)
- "완료 시 즉시 생성"은 실제 운영의 "리드타임 전 미리 생성"과 다름 → 이번엔 **로직·날짜계산 검증**이 목적이며, 실제 생성 시점(스케줄러)은 서버 몫임을 폼 안내/로그에 명시.
- 만기일을 endDate로 겸용 → 일반 마감과 갱신 만기의 의미가 한 필드에 겹침. 서버 모델 확정 시 분리 검토(이팀장과 §2 계약 시 논의) — 결정사항에 보류로 남김.
- 안내일·생성일은 현재 표시·계산용이며 별도 저장 안 함(다음 회차 endDate만 저장). 필요 시 후속.
