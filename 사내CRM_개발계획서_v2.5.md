# 사내 고객관리 + 할 일 통합 시스템 개발계획서 (v2.5)

> 작성일: 2026-05-30
> 대상: 사내 20인 내부용 (대량 배포 아님)
> 성격: 할 일 관리 + 고객(CRM) + 상품별 청약 파이프라인 + 반복/습관 + 다활동 통합 관리 웹앱
> 변경: v2.4 대비 파일 저장 방식 확정 — **구글 드라이브 백엔드(각자 본인 구글 계정 폴더), 외부 공유 절대 금지**. 화면은 투두 첨부, 실물은 구글 드라이브.

---

## 0. 한 줄 정의

**"고객마다 할 일이 붙고, 청약이 단계별 마감 시계로 진행되며, 실천 활동량이 누적되는 사내 20인용 영업 운영 도구."**

---

## 1. 배경 및 확정된 전제

| 항목 | 결정 | 설계 영향 |
|------|------|-----------|
| 기존 고보험 연동 | **불필요. 링크로 대체** | 데이터 동기화 개발 제거 |
| 배포 범위 | **사내 내부용 20인** | 앱스토어 불필요, 소규모 인프라 |
| 보험업법 | **비해당으로 판단** | 모집·수수료 규제 모듈 불필요 |
| 개인정보보호 | **핵심 과제. 보안 강화** | 8장이 가장 중요 |
| 사용 환경 | **PC + 모바일 둘 다** | 반응형 웹앱 단일 구현 |
| 알림 | **PC·모바일 동시 발송** | 별도 푸시 인프라 필요 (4장) |

> 주의: "보험업법 비해당"은 이아이 님 판단을 따른 것이나, 고객 개인정보를 다루는 이상 **개인정보보호법(PIPA)은 내부용이라도 적용**된다.

## 2. 핵심 데이터 모델

```
[직원(User)] ──1:N──> [할 일(Task)] <──N:1── [고객(Customer)]
     │                                            │
     └──> [청약 진행(Pipeline)] <──1:1── (고객당 1개 진행)
                  │
                  └──> [단계(Stage)] ──> [지연 기록(Delay)]
     │
     └──> [기준시간 설정(SLA Config)]  ← 사용자가 수정·저장
     └──> [알림(Notification)] ──> [기기 토큰(Device)]
     └──> [활동 로그(ActivityLog)]
```

### 2.1 고객 / 할 일 / 직원

- **고객**: 이름🔴, 연락처🔴, 상품, 상태, 갱신일, 담당직원, 메모🔴, 수신동의(2.4-c)
- **직원**: 이름, 로그인ID, 권한(대표/팀장/팀원), 비밀번호해시
- **할 일(Task)** — 반복·습관·알람을 모두 포괄하는 통합 구조:

| 필드 | 설명 |
|------|------|
| customer_id | 연결 고객 (선택, 없으면 일반 업무) |
| title | 제목 |
| cat | 카테고리 (미선택 시 **'기본'**) |
| time_type | **deadline(마감형)** / **range(구간형)** |
| end_date, end_time | 마감 날짜·시간 (공통) |
| start_date, start_time | 시작 날짜·시간 (구간형만) |
| repeat | 반복: none/daily/weekly_days/weekly/monthly/yearly (3장) |
| repeat_days | 요일 지정 시 선택 요일 |
| auto_regen | 완료 시 자동 재생성 여부 (3.4) |
| reminders[] | 알람 배열, 최대 5 (4장) |
| tags[] | 태그 (검색용) |
| category_id | **카테고리(계층형, 2.1-b)**, 미선택 시 '기본' |
| share[] | **공유 대상·권한 (2.1-c)** |
| files[] | **첨부 파일 (2.1-d)** |
| done, done_at | 완료 여부·시점 |
| streak | 연속 달성 (반복 항목, 습관 뷰용) |

> **설계 핵심**: 습관을 별도 테이블로 두지 않고 **"반복이 켜진 할 일 = 습관"**으로 통합. 데이터가 하나로 합쳐지고, 습관 뷰는 반복 항목만 모아 streak를 보여주는 화면.
> 🔴 = 개인정보. 8장 보안 대상. 시간 입력은 전부 **15분 단위 드롭다운**.

### 2.1-b 계층형 카테고리 (Category) — 신규 ★
> 영업뿐 아니라 사용자의 여러 활동(BNI·SLA·보험대리점·보험설계사·IT/다이렉트 등)을 한 시스템에서 관리. **부모(활동) → 자식(세부)** 2단 트리.

| 필드 | 설명 |
|------|------|
| id | 고유 ID |
| name | 이름 (예: "BNI 활동", "보험설계사 업무") |
| parent_id | 부모 카테고리 (없으면 최상위 활동) |
| color | 색상 (부모에만, 자식은 상속) |

- 사용자가 **직접 추가·수정·삭제** (활동/세부 모두)
- 기존 보험 5종(장기·화재·자동차신규·자동차갱신·보상)은 "보험설계사 업무" 부모 아래 자식으로 편입
- '기본'은 삭제 불가 기본 카테고리 (미선택 시 자동)
- 2단 트리 권장 근거: 인지 한계(Miller's Law) — 수십 개 평면 목록보다 부모-자식이 관리 용이

### 2.1-c 공유 (Share) — 신규 ★
| 필드 | 설명 |
|------|------|
| task_id | 공유 대상 투두 |
| scope | private(나만) / selected(지정) / team(팀 전체) |
| shared_with[] | (지정 시) 협업자 ID 목록 |
| permission | 협업자별 view(보기) / edit(편집) |

> 자동차 업무 흐름: 업무실장 ↔ 영업자가 같은 투두를 공유해 협업. 누가 무엇까지(보기/편집/완료 체크) 할지 권한 구분.

### 2.1-d 첨부 파일 (Attachment) — 신규 ★ (구글 드라이브 백엔드)

> **저장 방식 확정**: 파일 실물은 자체 서버가 아닌 **각 사용자 본인 구글 드라이브 폴더**에 저장. 시스템 DB에는 파일 자체가 아니라 **구글 파일 ID·위치만** 기록. 화면은 투두 첨부처럼 보이고 실물은 구글에 있음.

| 필드 | 설명 |
|------|------|
| task_id | 소속 투두 |
| name, type | 파일명, 유형(이미지/PDF) |
| size | 용량 |
| uploaded_by | 업로더 |
| uploaded_at | 업로드 시각 |
| gdrive_file_id | **구글 드라이브 파일 ID** (실물 위치) |
| gdrive_folder | 저장된 본인 구글 폴더 |

**동작 흐름**
```
[투두 화면: 업로드]──API──>[본인 구글 드라이브 폴더에 저장]
        │
        └── DB엔 구글 파일 ID만 기록 (파일 실물 X)
[다운로드] 투두에서 클릭 → 시스템 경유로 구글에서 가져옴
```

**채택 근거 (20인 내부용)**: 이미 클라우드(원드라이브)로 데이터 관리 중이고 **각자 본인 구글 계정 폴더에서 관리** → 개인정보 통제권이 각 사용자에게 있어 적합. 별도 스토리지 비용 없음, 구글이 백업·이중화.

> ⚠️ **핵심 원칙: 구글 공유 절대 금지**. 폴더·파일을 구글에서 "외부 공유"하면 시스템 권한과 무관하게 노출됨. 사용자에게 반드시 주지(8.4). 협업 공유는 **시스템 안에서만**(2.1-c) 이뤄지고, 구글 단에서는 본인만 접근.

| 필드 | 설명 |
|------|------|

| done_at | 완료 시점 |

|------|------|

### 2.4-b 안내 기록 (Notice Log) — 신규 ★ (수동 발송 기록용)

> 현재는 영업자가 직접 문자·카톡을 보내고, 시스템에는 **보냈다는 기록만** 남긴다. 추후 자동 발송(Phase 3)의 기반이 된다.

| 필드 | 설명 |
|------|------|
| stage_id | 어느 단계에서 안내했나 |
| done | 안내 여부 (체크) |
| sent_at | 안내 일시 (자동 기록) |
| channel | 안내 수단 (문자 / 카톡 / 전화) |
| memo | 보낸 내용 요약 → 추후 자동발송 템플릿으로 전환 |

### 2.4-c 고객 수신동의 (Consent) — 신규 ★ (자동발송 대비)

> 광고성 정보 자동 발송 시 정보통신망법상 **사전 수신동의가 필수**. 지금부터 기록해두면 Phase 3 전환이 매끄럽다.

| 필드 | 설명 |
|------|------|
| customer_id | 어느 고객인가 |
| sms_consent | 문자 수신동의 (동의/거부/미확인) |
| kakao_consent | 카톡 수신동의 (동의/거부/미확인) |
| consent_date | 동의 받은 날짜 |

### 2.5 기준시간 설정 (SLA Config) — 신규 ★

> 코드에 고정하지 않고 **화면에서 수정·저장**. 상품별로 따로 조정 가능. 아래는 초기값.

청약/처리 시작 시 **상품 유형을 선택하면 해당 템플릿이 자동 적용**된다. 총 5개 템플릿.

```
[고객] ──> 유형 선택
   ├── 장기보험        → 8단계
   ├── 화재보험        → 6단계
   ├── 자동차(신규)    → 5단계
   ├── 자동차(갱신)    → 4단계 (만기 60일 역산)
   └── 보상청구        → 7단계
```

#### ① 장기보험 (8단계)
| # | 단계 | 기준시간(초기값) |
|:---:|------|:---:|
| 1 | 고객 DB 수집 | 시작점 |
| 2 | 고객 연락·니즈 파악 | 수집 후 1일 |
| 3 | 개인정보 수령 → 보장분석 | 연락 후 2일 |
| 4 | 기본설계 + 고객 미팅 | 분석 후 3일 |
| 5 | 수정설계·제안 | 미팅 후 2일 |
| 6 | 청약·고지(심사) | 제안 후 2일 |
| 7 | 증권 전달 | 청약 후 5일 |
| 8 | 감사 인사·사후관리 | 전달 후 1일 |

#### ② 화재보험 (6단계)
| # | 단계 | 기준시간(초기값) |
|:---:|------|:---:|
| 1 | 고객 DB 수집 | 시작점 |
| 2 | 고객 연락·목적물 확인 | 수집 후 1일 |
| 3 | 물건 정보 수령 → 요율 산출 | 연락 후 1일 |
| 4 | 설계·제안 | 산출 후 1일 |
| 5 | 청약·증권 발행 | 제안 후 1일 |
| 6 | 증권 전달·감사 인사 | 청약 후 2일 |

#### ③ 자동차보험 (신규, 5단계)
| # | 단계 | 기준시간(초기값) |
|:---:|------|:---:|
| 1 | 고객 DB 수집 | 시작점 |
| 2 | 고객 연락·차량정보 수령 | 수집 후 1일 |
| 3 | 견적·비교 제안 | 연락 후 1일 |
| 4 | 청약·결제 | 제안 후 1일 |
| 5 | 증권 전달·감사 인사 | 청약 후 1일 |

#### ④ 자동차보험 (갱신, 4단계) ★ 만기일 역산
| # | 단계 | 마감 기준 | 내용 |
|:---:|------|:---:|------|
| 1 | 할인 활동 안내 | **만기 D-60** 자동 시작 | 무사고·마일리지 등 할인 요건 사전 안내 |
| 2 | 비교설계 및 안내 | **만기 D-30** | 갱신 견적 비교·설계 |
| 3 | 재계약 | **고객 요청일**(유동) | 고객이 원하는 시점에 체결 |
| 4 | 증권 안내 | 재계약 후 1일 | 증권 전달·확인 |

> 1·2단계는 만기일 역산, 3단계는 고객 요청일 입력, 4단계는 앞 단계 완료 기준. **만기일 자체는 절대 마감(하드 데드라인)**으로 별도 강한 경고 — 넘기면 무보험 공백 위험.

#### ⑤ 보상청구 (7단계) ★ 사후 처리
| # | 단계 | 기준시간(초기값) |
|:---:|------|:---:|
| 1 | 사고·청구 접수 | 시작점 |
| 2 | 필요 서류 안내 | 접수 후 1일 |
| 3 | 서류 수령·확인 | 안내 후 3일 |
| 4 | 손해사정·심사 | 서류 후 3일 |
| 5 | 보험사 청구 접수 | 사정 후 1일 |
| 6 | 지급 확인 | 청구 후 5일 |
| 7 | 결과 안내·종결 | 지급 후 1일 |

> 위 값은 모두 초기 예시이며, 운영하며 상품별로 직접 조정한다.

## 3. 청약 진행 파이프라인 (핵심 신규 ★★★★★)

### 3.1 동작 원리

```
[3단계 보장분석 완료]──(완료)──>[4단계 시계 시작]
   5/23 완료            +기준1일      → 5/24 마감 카운트다운
```

- 한 단계 완료 → **다음 단계 마감일 자동 산출** (완료시점 + 기준시간)
- 마감 시계가 항상 "현재 단계"에만 돌아감 (순차 의존)
- 전체 진행률(1/10 ~ 10/10) 표시

### 3.2 마감 초과 처리 ★

| 상황 | 처리 |
|------|------|
| 마감 임박 | PC·모바일 동시 알림 (사전 예고) |
| **마감 초과** | **강한 푸시** (반복·강조 알림) + "지연" 상태 표시 |
| **상대 사유로 지연** | **연장 마감 기능**: 사유 + 연장기간 입력 → 마감 재설정 |

> 연장은 "그냥 미루기"가 아니라 **사유와 기간을 기록**해야 활동량 분석에서 추적된다.

### 3.3 병목 분석 (근거 체크) ★

각 단계의 **실제 소요 시간 vs 기준 시간**을 누적 비교한다.

```
단계별 평균 소요일 (예시)
2.고객연락     ▓ 0.8일  (기준 1일, 양호)
3.보장분석     ▓▓▓ 2.9일 (기준 2일, 초과 ← 병목)
5.고객미팅     ▓▓ 2.1일  (기준 3일, 양호)
8.청약         ▓▓▓▓ 3.5일 (기준 2일, 초과 ← 병목)
```

- "어느 단계에서 자주 막히는가"를 데이터로 파악
- 지연 사유를 함께 집계 → 고객 탓 / 내부 탓 구분
- 개선 의사결정의 객관적 근거 제공

## 4. 알림 기능 (PC·모바일 동시) ★★★★

### 4.1 동시 발송 원리

```
[서버: "5단계 마감 2시간 전!"]
        ├──> PC 브라우저 (웹 푸시)
        └──> 모바일 (웹 푸시 / 앱 푸시)
```

- 한 사용자의 **여러 기기 토큰을 모두 저장** → 알림 시점에 전부 동시 발송
- 기기 등록/해제 관리 (퇴사·기기변경 대응)

### 4.2 알람 설정 (항목당 최대 5회) ★

각 할 일에 알람을 **최대 5개**까지 설정. 알람마다 두 가지를 정한다.

**① 시점** (마감 기준 얼마 전)
- 기본 옵션: 정시 / 15분 / 30분 / 1시간 / 1일 전
- **사용자 지정**: 분 단위 직접 입력 (예: 45분, 3시간)

**② 알람 시 동작** (3종)
| 동작 | 의미 |
|------|------|
| **작업완료** | 알람에서 버튼 눌러 즉시 완료 처리 (※ 자동 아님 — 사용자 확인 필수) |
| **연장** | 사유·기간 입력해 마감 재설정 (3.2 연장과 연결) |
| **알람만 체크** | 확인만, 상태 변화 없음 |

- 마감 초과 시 **강한 반복 푸시** (3.2와 연결)
- 모든 알람은 PC·모바일 동시 발송 (4.1)

> **작업완료 자동화 금지 원칙**: 알람이 울렸다고 시스템이 자동 완료하면 안 한 일이 완료로 기록될 위험. 반드시 사용자가 버튼을 눌러야 완료된다.

### 4.3 주의
- 웹 푸시는 사용자가 브라우저에서 "알림 허용" 1회 필요
- 모바일 절전 시 지연 가능 → **앱 내 알림 + 푸시 이중 구조** 권장

## 5. 캘린더 · 할 일 UI (★★★★ 통합 명세)

프로토타입으로 검증 완료한 화면 구성. 좌측 목록 + 우측 캘린더(일간/주간 전환) 구조.

### 5.1 작업 시간 타입 — 2종
| 타입 | 입력 | 표시 | 조작 |
|------|------|------|------|
| **종일** | 날짜만 | 상단 종일 칸 | - |
| **기간형** | 시작·종료 시간 | 길이 있는 막대(타임바) | 가운데 끌어 이동 / 양끝 끌어 길이 조절 |
| **마감형** | "○시까지" | 점 마커 | 끌어서 마감 시각 이동 |

> 시간 조작은 **15분 단위 스냅**, 운영시간(예: 8~20시) 경계 제한.

### 5.2 뷰 전환 — 4종
- **일간**: 시간 격자 + 막대/마커. 시간 드래그. 좌우 화살표로 날짜 이동.
- **주간**: 7일 칸. 칩 드래그로 요일 이동. 요일명 클릭 → 일간 진입.
- **월간**: 달력 그리드. 날짜 칸 더블클릭 → 새 할 일. 칸 초과 시 "+N" 압축. ● 마감형 / ▭ 구간형 기호.
- **연간**: 12개월 카드. 월별 일정 수 + 카테고리 색 점 요약. 카드 클릭 → 해당 월. (영업 관점: 갱신·청약이 어느 달에 몰리는지 파악)
- (Phase 3) **외부 캘린더 가져오기**: 구글 캘린더(OAuth), 카카오 캘린더(정책 확인 필요). 우선 **읽기 전용 가져오기**부터 — 양방향 동기화는 충돌 처리가 복잡하므로 후순위.

### 5.3 드래그 이동 — 2종
- **목록 내 순서 변경**
- **목록 → 캘린더 / 캘린더 → 다른 요일** 날짜 이동
- (일간) 시간 막대·마커 끌어 시각 변경

### 5.4 마감 겹침 묶음 ★
- 같은 시간대(기본 **30분 이내**, 설정 가능) 마감이 겹치면 **"○○○ 외 N개"**로 압축
- **호버** → 겹친 항목 리스트 펼침
- 리스트 항목 **클릭** → 상세 팝업

### 5.5 보기 제어
- **완료 작업 표시/숨기기** 토글 (목록·캘린더 동시 적용)
- **카테고리별 필터**: 기본·장기·화재·자동차신규·자동차갱신·보상 색상 구분, 켜고 끄기
- **반복 항목 표시/숨기기** 토글 (매일 반복이 캘린더를 채우지 않도록)

### 5.6 항목 수정 · 태그 · 검색
- **클릭/더블클릭 → 편집 폼**: 제목·고객·카테고리·유형(마감/구간)·날짜·시간·내용·태그·반복·알람 전부 수정. 삭제 가능.
- **카테고리 '기본'**: 미선택 시 자동 적용 (회색)
- **태그**: 자유 입력(Enter 추가). 자주 쓰는 태그 자동완성으로 표기 통일 유도 권장.
- **검색**: 제목·고객명·태그를 한 번에 검색. 태그 칩 클릭으로 즉시 필터.

### 5.7 기술 주의사항 (PC + 모바일 공존) ★
| 항목 | 처리 |
|------|------|
| 시간 드래그 | **Pointer Event** 사용 → PC 마우스·모바일 터치 단일 코드 |
| 목록/요일 드래그 | HTML5 DnD는 모바일 미지원 → **터치 드래그 라이브러리** 병행 |
| 호버 미리보기 | 모바일은 호버 없음 → **탭하면 바로 리스트/팝업** 분기 |
| 마감 마커 겹침 | 같은 시각 다수 → 묶음(5.4)으로 해결 |

> 가장 흔한 실수: PC에서만 테스트하고 모바일 드래그·호버를 빠뜨리는 것. 두 환경 모두 사용하므로 설계 단계부터 분기 처리.

## 5-B. 반복 · 습관 · 자동 재생성 (★★★★ 신규)

### 5-B.1 반복 5종
모든 할 일에 반복 설정 가능. 반복이 켜진 항목 = 습관으로 취급.

| 반복 | 의미 |
|------|------|
| 반복 안 함 | 일회성 |
| 매일 | 매일 |
| 요일 지정 | 월·수·금 등 선택 요일 |
| 매주 | 같은 요일 주 단위 |
| 매월 | 같은 날짜 월 단위 |
| 매년 | 같은 날짜 연 단위 |

- 반복 항목은 해당 주기마다 캘린더에 자동 등장 (5.5 토글로 표시 제어)
- 습관 뷰: 반복 항목만 모아 **연속 일수(streak)** + 주간 달성 그리드 표시

### 5-B.2 습관 화면
- 오늘 해당 습관만 모아 체크 → streak 증감
- 주간 달성 그리드 (✓완료 / 미완료 / 예정 / 해당없음)
- 더블클릭 → 수정 (반복·시간·알람)
- 마감형/구간형 2종 동일 적용

### 5-B.3 완료 시 자동 재생성 (갱신용) ★★★
> 일반 반복(시간 트리거)과 달리, **완료가 다음 발생을 낳는 연쇄**. 매주·매월·매년에서만 활성.

**핵심: 생성일 ≠ 안내일 ≠ 만기일 (3단 분리)**

```
[투두 생성] ───리드타임──> [안내 목표] ───────> [만기일]
 만기 67일 전              만기 60일 전          하드 데드라인
 (준비 시작)               (실제 발송)          (절대 마감)
```

| 설정 | 내용 |
|------|------|
| 안내 목표 시점 | 만기 며칠 전 안내 (30/60/90 + **사용자 지정**) |
| 준비 리드타임 | 안내일보다 며칠 먼저 투두 생성 (1/3/7/14 + **직접 입력**) |
| 생성 시점 | = 만기 (안내목표 + 리드타임)일 전 자동 계산 |

> **왜 분리하나**: 투두가 안내 당일 생성되면 준비할 시간이 없다. 리드타임만큼 미리 생성돼야 다음 날 여유 있게 발송 가능. (자동차 갱신: 안내 60일 전 + 리드타임 7일 → **67일 전 생성**)

### 5-B.4 빼앗긴 고객 처리 (운영 중 결정) ★
타사 이탈 시 3가지 중 선택 — **업무하며 케이스별로 결정**:

| 정책 | 동작 |
|------|------|
| **매번 묻기** (기본) | 이탈 표시 시 "내년에도 안내?" 확인 |
| **유지(재안내)** | 삭제 안 함 → 내년 갱신 시점 재접근 |
| **영구 삭제** | 갱신 라인 종료, 재생성 중단 |

> 자동 재생성 안전장치: 재생성 첫 단계에 "아직 우리 고객인가" 확인, 일정 횟수 이상 미완료 시 재생성 자동 중단 (유령 일정 방지).

## 7. 기능 범위 및 로드맵

### Phase 1 — MVP
- [ ] 로그인 / 권한
- [ ] 고객 등록·조회 (+ 수신동의 필드 기록)
- [ ] 할 일 통합 모델: 마감형/구간형, 15분 드롭다운
- [ ] **계층형 카테고리** (부모-자식, 사용자 추가·관리)
- [ ] 캘린더 4종 뷰 (일·주·월·연)
- [ ] 드래그 이동 (목록 순서 / 요일 이동 / 일간 시간 드래그) — **Pointer Event 기반**
- [ ] 항목 수정·태그·검색, 카테고리 필터, 마감 겹침 묶음, 완료 표시/숨김
- [ ] **파일 첨부** (이미지·PDF 업로드/다운로드, 업로더 기록)
- [ ] 청약 파이프라인 5종 — 단계 완료 → 다음 마감 자동
- [ ] 단계별 안내 여부 체크 (수동 발송 기록)

### Phase 2 — 운영
- [ ] **투두 공유·협업** (범위 3종 + 보기/편집 권한)
- [ ] PC·모바일 동시 알림 + 알람 최대 5회(사용자 지정) + 3종 동작
- [ ] 마감 초과 강한 푸시 + 연장(사유·기간) 기능
- [ ] 반복 5종 + 습관 뷰(streak·주간 그리드)
- [ ] 완료 시 자동 재생성 (생성일/안내일 분리·리드타임) + 빼앗긴 고객 처리
- [ ] 자동차 갱신 만기일 역산 + 만기 하드 데드라인 경고
- [ ] 기준시간 설정 화면 (상품별, 사용자 저장 방식)
- [ ] 활동량 누적 + 단계 병목 분석

### Phase 3 — 확장
- [ ] **외부 캘린더 가져오기** (구글·카카오, 읽기 전용 우선)
- [ ] **전산 자동 문자 발송** (만기 감지 → 자동 안내, 수신동의 기반)
- [ ] 고보험 등 외부 **링크 바로가기**
- [ ] 데이터 엑셀 내보내기
- [ ] 모바일 푸시 고도화

### 보류 (추후 결정)
- 메인 대시보드 (인생 철학·목표·이달/오늘 중요사항)
- 감사일기
> 이유: "있으면 좋은" 개인 관리 영역. 영업 핵심(파이프라인·캘린더·공유·첨부)이 당장 돈이 되는 우선순위. 첫 출시 범위가 무한정 커지지 않도록 분리.

> 안내 방식 전환: **Phase 1 = 사람이 보내고 기록만 → Phase 3 = 시스템 자동 발송**. Phase 1의 "안내 메모"가 Phase 3의 발송 템플릿이 되고, 수신동의 기록이 발송 자격 판단 근거가 된다.

## 8. 개인정보보호 · 보안 강화 (핵심 ★★★★★)

### 8.1 저장 보안
| 항목 | 조치 |
|------|------|
| 서버 위치 | **국내 서버** 보관 |

### 8.2 접근 통제
- 대표=전체 / 팀장=팀 / 팀원=본인 고객
- **서버(API)에서도 권한 재검증**

- **공유 투두**: 공유 범위가 곧 정보 노출 범위. 고객 파일이 있는 투두는 "팀 전체 공유" 신중, **지정 공유(필요한 사람만)** 기본 권장. 보기 전용/편집/다운로드 권한 분리.

### 8.3 감사 로그
- 누가·언제·어떤 고객정보를 조회/수정/삭제했는지 기록
- 대량 조회·내보내기 별도 기록
- **첨부 파일 다운로드 로그**: 누가 어떤 청약서·증권을 받아갔는지 기록 (사고 추적 근거)

### 8.4 첨부 파일 보안 ★ (구글 드라이브 백엔드)
- 파일 실물은 **각 사용자 본인 구글 드라이브 폴더**에 저장 (개인정보 통제권 = 각 사용자)
- **구글 외부 공유 절대 금지** — 사용자 교육·시스템 안내로 반복 주지. "링크 있는 모든 사용자" 설정 금지, 본인 계정만 접근.
- 협업 공유는 **시스템 내부에서만**(2.1-c) — 구글 단에서는 공유하지 않음
- 다운로드는 시스템 경유 → **다운로드 로그 기록**(8.3). 구글 직접 링크 배포 금지(로그 추적 끊김)
- 다운로드 권한은 시스템 공유 권한과 연동 (보기 전용자의 다운로드 가부는 정책 결정)
- 구글 계정 자체 보안(2단계 인증 등) 권장

## 10. 리스크 및 자기비판

| 리스크 | 내용 | 대응 |
|--------|------|------|
| "내부용이라 보안 불필요" 착각 | 가장 흔한 사고 원인 | 8장 출시 전 필수 |
| 푸시 알림 미수신 | 절전·권한 거부로 누락 | 앱 내 알림 이중화 |
| 연장 기능 남용 | 모든 지연을 "고객 탓"으로 | 사유 집계로 패턴 감시 |
| 기준시간 비현실 | 초기값이 실무와 동떨어짐 | 사용자 설정·운영중 조정 |
| **자동 발송 시 법 위반** | 수신동의 없는 광고성 발송 = 정보통신망법 위반 | 수동 단계부터 동의 기록(2.4-c), Phase 3 발송 전 동의 검증 |
| **갱신 만기 공백** | 고객 요청일이 만기 초과 시 무보험 | 만기일 하드 데드라인 별도 강경고 |
| **모바일 드래그·호버 누락** | PC만 테스트 시 모바일에서 시간 이동·미리보기 불가 | Pointer Event 통일, 호버는 탭으로 대체 (5.7) |
| **자동 재생성 유령 일정** | 이탈 고객에게 갱신 투두 계속 생성 | 재생성 첫 단계 고객 확인 + 미완료 누적 시 중단 (5-B.4) |
| **알람 작업완료 오기록** | 자동 완료 처리로 안 한 일이 완료됨 | 작업완료는 사용자 버튼 클릭 필수 (4.2) |
| **반복 항목 캘린더 과밀** | 매일 반복이 달력을 채움 | 반복 표시/숨김 토글 (5.5) |
| **공유 범위 과다 노출** | 팀 전체 공유로 고객 증권 노출 | 지정 공유 기본, 권한 분리 (8.2) |
| **첨부 파일 유출** | 구글 외부 공유로 청약서·증권 노출 | **구글 공유 금지** 주지, 시스템 경유 다운로드+로그 (8.4) |
| **구글 의존 리스크** | 구글 장애·접속 불가 시 파일 접근 불가 | 폐쇄망 불필요한 내부용이라 수용, 중요 파일 별도 백업 권장 |
| 범위 확대(scope creep) | 인생 OS로 비대화 → 출시 지연 | 대시보드·감사일기 보류, 영업 핵심 우선 |
| 기능 과욕 | 동시 욕심 | Phase 1 MVP 집중 |
| 단독 개발 의존 | 담당자 부재 시 마비 | 문서·코드 공유 |
